# PT Biz Tools — UI Refactoring & Integration PRD

## Project Overview
PT Biz Tools is a Next.js + Express/Prisma SaaS platform for physical therapy business coaching. The frontend (ptbiztools-next) contains multiple tool pages including a Discovery Call Grader, Danny's Financial Audit, Compensation Calculator, and clinic analytics. The project recently underwent a rescue/cleanup phase and now needs focused UI refactoring work.

## Tech Stack
- Frontend: Next.js (App Router), React, TypeScript, Tailwind CSS + custom CSS
- Backend: Express.js, Prisma ORM, PostgreSQL (Railway)
- AI Integration: Anthropic Claude API (backend)
- Deployment: Vercel (frontend), Railway (backend)

## Task 1: Rewrite DiscoveryCallGrader.tsx — Monolith Decomposition

### Problem
`ptbiztools-next/src/components/discovery/DiscoveryCallGrader.tsx` is a 1081-line monolith (42KB). It contains random UI boxes from failed AI agent attempts, mixed concerns (state management, API calls, UI rendering, file handling, PDF generation), and needs complete decomposition into clean, maintainable components.

### Requirements

#### 1.1 Analyze Current Monolith Structure
- Map all state variables, hooks, and their dependencies
- Identify distinct UI sections (upload area, progress/diagnostics, results display, scoring)
- Catalog all API calls and data flows
- Document which sections are "random UI boxes from failed agent attempts" that need removal

#### 1.2 Extract Shared Types and Constants
- Create `types.ts` for grading types, API response types, score types
- Create `constants.ts` for PHASES, RED_FLAGS, scoring weights, and other config
- Move from inline definitions to importable modules

#### 1.3 Extract Custom Hooks
- `useGrading.ts` — grading state machine (idle → uploading → grading → complete → error)
- `useFileUpload.ts` — file selection, validation, transcript extraction
- `useGradingResults.ts` — results state, score calculations, phase breakdowns

#### 1.4 Decompose into Sub-Components
- `FileUploadPanel.tsx` — drag-and-drop file upload with validation
- `GradingProgress.tsx` — live progress UI with staging and diagnostics
- `PhaseCard.tsx` — expandable phase scoring card with rubric (great/mistakes)
- `RedFlagsPanel.tsx` — red flag checkboxes with deductions
- `ScoreSummary.tsx` — visual score breakdown with tier display
- `GradingTabs.tsx` — tab navigation (Grading | Feedback | Summary)
- `FeedbackView.tsx` — detailed feedback display
- `SummaryView.tsx` — overall summary with PDF export

#### 1.5 Clean Up Bad UI
- Remove all "random UI boxes" from failed agent attempts
- Remove any dead code, unused state variables, commented-out sections
- Ensure consistent styling with the project's design system

#### 1.6 Reassemble Main Component
- `DiscoveryCallGrader.tsx` should become a thin orchestrator (~100-150 lines)
- Import and compose sub-components
- Maintain all existing functionality (upload, grade, display results, PDF export)

### Success Criteria
- DiscoveryCallGrader.tsx reduced from 1081 lines to ~150 lines
- All sub-components individually testable
- No functionality regression — upload, grading, results, PDF all still work
- Clean removal of failed agent UI artifacts
- Consistent with project design system

## Task 2: Danny's Modal Integration (Merge Danny's UI into Production Architecture)

### Problem
Danny (a team member) built a version of the Discovery Call Grader with excellent UI (tabbed interface, phase cards, red flags, manual scoring) but bad architecture (direct Anthropic API calls from browser, synchronous PDF encoding, data URIs). The production version has good architecture (backend API, proper file handling, session tracking) but different/inferior UI. These need to be merged: Danny's UI + Production's backend.

### Reference Document
See `ptbiztools-next/PLAN_DANNY_INTEGRATION.md` for full analysis.

### Requirements

#### 2.1 Keep Production Backend Integration
- Continue using `gradeDannySalesCallV2` API endpoint
- Keep file upload via `extractTranscriptFromFile`
- Keep session logging via `logAction`
- Keep PDF generation via `generatePDF` utility

#### 2.2 Adopt Danny's UI Components
- Implement tabbed interface: Grading | Feedback | Summary tabs
- Implement phase cards with expandable rubric showing great points and mistakes
- Implement red flags section with checkbox list and score deductions
- Implement summary view with visual score breakdown
- Implement manual scoring sliders for override capability

#### 2.3 Fix Danny's Critical Errors
- Replace direct Anthropic API calls with backend API integration
- Fix PDF encoding: use FileReader.readAsDataURL instead of synchronous loop
- Fix report download: use Blob + URL.createObjectURL instead of data URIs
- Add proper error handling with try/catch and user feedback

#### 2.4 Merge Styling
- Convert Danny's inline styles to CSS classes
- Match existing design system (grader-page, tool-page patterns)
- Ensure responsive layout works on all screen sizes
- Integrate with existing danny-tools.css and danny-theme.css

#### 2.5 Integration Testing
- Verify backend API grading still works end-to-end
- Verify file upload handles PDF, TXT, and DOCX formats
- Verify PDF report generation and download works
- Verify tab navigation and state persistence
- Verify phase card expand/collapse functionality
- Verify red flag checkbox scoring deductions

### Success Criteria
- Danny's tabbed UI is live in production
- All grading goes through backend API (no browser-side Anthropic calls)
- PDF generation works without browser freezing
- Report download works for files > 2MB
- All existing functionality preserved
- Responsive on desktop and tablet
